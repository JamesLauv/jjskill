# 文档到决策材料共创

使用这个 Skill 时，目标不是简单总结文档，而是把原始材料转化为可以用于沟通、评审、敲定方案和推进执行的决策材料。

## 核心原则

除非用户明确要求“直接生成”或“不要沟通”，否则不要从源文档直接跳到最终产物。

默认应先帮助用户完成三件事：

1. 看清原文里真正的事实、方案、风险和待确认问题。
2. 通过交互补充原文没有写出来的业务判断、组织逻辑和沟通策略。
3. 把材料重构成可决策、可沟通、可推进的结构。

好的输出不只是“整理得清楚”，还应该帮助用户把事情往前推动。

## 默认工作流

1. **读取原文，还原事实**  
   保留原文明确写到的方案、条件、风险、待确认点，不急着美化。

2. **分析结构和核心矛盾**  
   提炼方案分歧、决策链路、依赖关系、风险来源和利益相关方。

3. **与用户交互补充判断**  
   吸收用户补充的业务背景、内部口径、客户偏好、组织约束和隐含判断。

4. **确认使用场景和输出格式**  
   在进入最终产物制作前，确认材料是用于内部评审、客户沟通、技术讨论、会议推进还是正式沉淀，并确认输出格式。

5. **重构成决策材料**  
   把原始信息转成方案定位、优劣势、风险、核对清单、推荐路径、责任人和下一步动作。

6. **输出最终产物**  
   按用户确认的格式生成文件或内容，并在可能时完成基本校验。

## 原文信息拆解

读取文档后，先拆成以下几层：

- **事实**：原文明确说了什么。
- **方案**：有哪些路径、选项、环节或技术链路。
- **依赖**：哪个选择会影响后续选择。
- **风险**：技术风险、交付风险、成本风险、周期风险、责任边界风险、沟通风险。
- **待确认问题**：什么问题必须找谁确认。
- **利益相关方**：内部团队、产品、研发、销售、客户、合作方、管理层等。
- **决策压力**：哪些事情现在必须定，哪些可以后置。

要区分“原文事实”和“你的判断”。不要把推断伪装成原文结论。

## 第一轮分析输出

当用户要求分析、总结或梳理时，先输出结构化分析，再制作最终材料。

建议包括：

- 这份材料真正要解决的问题。
- 当前有哪些核心方案或路线。
- 方案之间的关键差异。
- 哪些是技术问题，哪些是沟通策略问题。
- 哪些信息还缺，需要向谁确认。
- 建议用什么结构去跟团队沟通。

如果用户已经明确要求生成文件，也可以先给简短分析，再进入产物制作。除非用户明确说“直接生成，不要解释”。

## 交互共创要求

用户补充的判断通常很关键，要主动吸收并重构方案。

以下类型的话，不要当作普通备注：

- “内部肯定会问到这个。”
- “这个方案对公司更好交代。”
- “客户其实更关心长期架构。”
- “这个不是技术问题，是沟通策略。”
- “这个要拿去跟同事确认。”
- “我要会上推动敲定。”
- “这个方案虽然技术好，但内部可能不好过。”
- “短期和长期要分开讲。”

遇到这类信息时，应重新调整方案定位、沟通顺序和最终产物结构。

## 输出格式确认

在进入最终产物制作前，先确认用户需要哪种输出格式，除非用户已经明确指定。

可选格式包括：

- **HTML**：适合汇报阅读、内部评审、视觉化方案展示。
- **drawio**：适合流程图、架构图、方案依赖关系图、会议中拖拽讨论。
- **Markdown / 飞书文档**：适合沉淀文字版方案、会议纪要、协作文档。
- **Excel**：适合会上勾选方案、记录备注、明确责任人、跟踪推进状态。
- **PPT / Word / 图片 / JSON Canvas / 其他格式**：按用户场景判断。

确认时不要只问“要什么格式”，而要结合用途给建议：

- 如果用户要“拿去开会推动敲定”，优先建议 **Excel + HTML**。
- 如果用户要“给领导快速看”，优先建议 **HTML 或 PPT**。
- 如果用户要“跟技术同事讨论链路”，优先建议 **drawio**。
- 如果用户要“沉淀成正式方案文档”，优先建议 **Markdown / 飞书文档**。
- 如果用户要“多人在线协同编辑”，优先建议 **飞书文档 / 飞书表格**。
- 如果用户已经明确指定格式，直接按指定格式输出，不再反复确认。

## 决策材料推荐结构

### 方案评估类

优先使用以下结构：

| 模块 | 作用 |
|---|---|
| 结论先行 | 说明这份材料要帮助大家决定什么 |
| 背景与目标 | 解释为什么现在要评估 |
| 方案定位 | 说明每个方案为什么存在 |
| 方案对比 | 对比能力、成本、风险、周期、责任边界 |
| 决策链路 | 展示哪些选择会影响后续选择 |
| 风险与阻塞 | 区分已知风险和待确认风险 |
| 核对清单 | 明确要找谁确认什么 |
| 推荐推进路径 | 给出下一步怎么推进 |

### 会议推进类

优先使用以下结构：

| 模块 | 作用 |
|---|---|
| 会上要定什么 | 避免会议变成泛泛讨论 |
| 建议讨论顺序 | 控制沟通节奏 |
| 可选方案 | 明确每个选项 |
| 每个方案优劣势 | 帮助团队比较 |
| 需要谁确认 | 把模糊问题变成责任人 |
| 会后动作 | 把结论转成推进任务 |

### 内部评审类

必须考虑：

- 哪个方案符合公司产品规划。
- 哪个方案最容易短期交付。
- 哪个方案最符合客户长期诉求。
- 哪个方案会带来隐藏交付或售后负担。
- 哪个方案技术上可行但组织沟通难。
- 哪个方案长期正确但短期成本高。
- 哪些内容需要先对内统一口径，再对外沟通。

## 各类产物要求

### HTML

用于可视化汇报和内部评审。

当用户选择 HTML 输出时，默认按以下硬规格执行，除非用户明确给出其他要求。

技术要求：

- 使用单文件 HTML。
- 包含完整 `<!DOCTYPE html>`、`html`、`head`、`body` 结构。
- 使用 Tailwind CSS CDN：`<script src="https://cdn.tailwindcss.com"></script>`。
- 使用原生 JavaScript 实现交互，尤其是 Tab 切换。
- 不使用 React。
- 不使用 npm。
- 不依赖构建工具。
- 保存为 `.html` 后可以直接双击浏览器打开。
- 如果用户要求“只输出 HTML 代码”，不要输出 Markdown 解释，只输出完整 HTML 文件代码。

页面风格：

- 类似 Claude Artifacts 的单页交互产物。
- 卡片式布局。
- 现代 SaaS 风格。
- 浅灰背景。
- 圆角卡片。
- 数据指标展示。
- Tab 切换。
- 流程图或决策链路展示。
- 响应式布局，兼顾桌面和移动端。

内容结构建议：

- 顶部结论区。
- 关键指标卡片。
- 方案卡片。
- Tab 切换区。
- 决策链路图或流程图。
- 方案对比矩阵。
- 核对清单。
- 推荐沟通策略。

HTML 输出质量要求：

- 不是把 Markdown 简单包进网页，而是重构成可阅读、可演示、可交互的页面。
- 首屏要能快速看出材料主题、核心结论和关键指标。
- Tab 内容要围绕不同沟通任务组织，例如“决策链路 / 方案矩阵 / 核对清单 / 沟通策略”。
- 流程图可以用 HTML/CSS 结构表达，不强制使用外部图库。
- 卡片、表格、流程节点之间要有清晰层级，避免信息堆叠。
- 文案要服务决策沟通，不要写成普通网页说明书。

### drawio

用于会议讨论和技术链路对齐。

建议包含：

- 方案泳道。
- 决策节点。
- 依赖箭头。
- 风险标注。
- 待确认问题区。
- 各方责任边界。

### Excel

用于会上敲定和会后跟踪。

建议工作表：

1. `会议总览`
2. `方案优劣势`
3. `环节方案勾选`
4. `核对清单`
5. `推进任务`

建议下拉字段：

- 会议选择：`待确认 / 已选 / 不选 / 需补充评估`
- 状态：`未开始 / 沟通中 / 已确认 / 有风险 / 关闭`
- 优先级：`高 / 中 / 低`

Excel 的重点不是视觉炫酷，而是能在会上直接使用。

### Markdown / 飞书文档

用于沉淀正式文字材料、会议纪要或协同文档。

建议包含：

- 背景
- 目标
- 方案对比
- 关键判断
- 风险
- 待确认问题
- 下一步动作

如用户需要飞书文档，优先使用相应 Lark skill / CLI 进行创建或更新。

## 质量判断标准

弱输出只是把原文排版变漂亮。

强输出应该做到：

- 找到真正需要决策的问题。
- 把短期落地和长期架构分开。
- 说明每个方案的组织定位。
- 把隐含沟通逻辑显性化。
- 把待确认问题分配到对象。
- 能直接用于会议讨论或推进。
- 能帮助用户说服团队、统一口径、形成结论。

## 直出模式与共创模式

如果用户明确要求“直接输出”，可以直接生成。

但直出模式应更贴近原文事实，不要过度引入战略判断。

如果后续用户要求“更适合沟通”或“更适合会议推进”，应生成共创版。

可以区分两类文件：

- **直出版**：基于原文结构，低解释、低重构。
- **共创版**：吸收用户判断，面向沟通、决策和推进。

## 推荐提问

只在必要时提问，优先问 1-3 个关键问题。

可选问题：

- “这份材料主要是给内部评审、客户沟通，还是项目推进会用？”
- “你希望优先强调短期落地，还是长期架构？”
- “有没有公司内部一定会被问到的产品规划或交付口径？”
- “哪些对象需要被说服：产品、研发、销售、客户、领导、合作方？”
- “最后你希望会议上敲定什么？”
- “你希望输出成 HTML、drawio、Markdown/飞书文档、Excel，还是其他格式？”

如果用户已经提供足够上下文，不要为了流程完整而多问，直接推进。

## 文件输出规则

创建文件时：

- 保存到用户指定目录。
- 使用清晰中文文件名。
- 必要时在文件名中标明用途，如“内部沟通版”“会议推进表”“直出测试”。
- 尽量校验文件存在、格式可解析或可打开。
- 最终回复只链接最终产物，并简要说明每个文件适合怎么用。

## 避免事项

不要：

- 只按段落总结原文。
- 把不确定信息包装成确定结论。
- 把短期 workaround 和长期架构混成一个选择。
- 忽略组织内部产品规划、交付口径、责任边界。
- 做出视觉好看但会议上不能用的材料。
- 在用户已经明确格式时反复确认格式。
- 为了流程完整而问太多问题。
